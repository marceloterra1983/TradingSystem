## Migration Guides

### Upgrading from v0.2.0 to v1.0.0

**Breaking Changes**: None

**New Features**:

- Service launch endpoint (`/launch`)
- Additional monitored services

**Steps**:

1. Stop service: `docker stop service-launcher`
2. Pull latest code: `git pull`
3. Update dependencies: `cd apps/status && npm install`
4. Restart: `docker start service-launcher`
5. Verify: `curl http://localhost:3500/health`

**Configuration Changes**:

- Optional: Add `SERVICE_LAUNCHER_USE_WT=true` for Windows Terminal support

---

### Upgrading from v0.1.0 to v0.2.0

**Breaking Changes**: None

**New Features**:

- Comprehensive health checks (`/api/health/full`)
- Caching

**Steps**:

1. Stop service
2. Update code
3. **Ensure health check script is executable**: `chmod +x scripts/maintenance/health-check-all.sh`
4. Restart
5. Test new endpoint: `curl http://localhost:3500/api/health/full`

**Configuration Changes**:

- Optional: `HEALTH_CHECK_TIMEOUT_MS=10000`
- Optional: `HEALTH_CHECK_SCRIPT_PATH=scripts/maintenance/health-check-all.sh`

## Release Process

### Semantic Versioning

Service Launcher follows [Semantic Versioning 2.0.0](https://semver.org/):

- **MAJOR** (v2.0.0): Breaking changes, incompatible API changes
- **MINOR** (v1.1.0): New features, backwards-compatible
- **PATCH** (v1.0.1): Bug fixes, backwards-compatible

### Release Checklist

- [ ] Update version in `package.json`
- [ ] Update CHANGELOG.md with changes
- [ ] Run tests: `npm test` (all passing)
- [ ] Run linting: `npm run lint` (no errors)
- [ ] Update documentation (if needed)
- [ ] Create git tag: `git tag v1.1.0`
- [ ] Push tag: `git push --tags`
- [ ] Build Docker image: `docker build -t service-launcher:v1.1.0 .`
- [ ] Deploy to staging and test
- [ ] Deploy to production
- [ ] Announce release (Slack, email)
- [ ] Monitor for issues (first 24h)
