## System Requirements

### Operating System

**Supported**:

- Linux (Ubuntu 20.04+, Debian 11+, CentOS 8+, RHEL 8+)
- Windows 10/11 (with WSL2 recommended)
- macOS 11+ (BigSur or later)

**Tested**:

- Ubuntu 22.04 LTS (primary development environment)
- Windows 11 + WSL2 (Ubuntu 22.04)

### Hardware Requirements

**Minimum**:

- CPU: 1 core
- RAM: 256 MB
- Disk: 50 MB (for service code and logs)

**Recommended**:

- CPU: 2 cores
- RAM: 512 MB
- Disk: 500 MB (includes logs, cache, and dependencies)

**Notes**:

- Service Launcher is lightweight and has minimal resource requirements
- Actual resource usage depends on number of monitored services
- Health check script may require additional CPU/memory

## External Service Dependencies

### Required Services

| Service | Purpose | Startup Order |
|---------|---------|---------------|
| None | Service Launcher has no hard dependencies | N/A |

**Note**: Service Launcher can start independently. It will report services as "down" if they're not running, but this doesn't prevent Service Launcher from operating.

### Monitored Services (Optional)

Service Launcher monitors these services (if configured):

| Service | Port | Required | Purpose |
|---------|------|----------|---------|
| Workspace API | 3200 | No | Workspace management |
| TP Capital | 4005 | No | Signal ingestion |
| Documentation API | 3401 | No | Documentation management |
| Firecrawl Proxy | 3600 | No | Web scraping |
| Dashboard | 3103 | No | UI frontend |
| Documentation Hub | 3400 | No | Documentation UI (docs-hub container) |
| Prometheus | 9090 | No | Metrics collection |
| Grafana | 3000 | No | Metrics visualization |
| QuestDB | 9000 | No | Time-series database |

**None of these are required** for Service Launcher to start. They're only needed if you want to monitor their health.

### Database Dependencies

**None**. Service Launcher does not use any database.

**Note**: Health check script may query TimescaleDB or QuestDB, but Service Launcher itself doesn't depend on them.

## Network Requirements

### Ports

**Inbound** (Service Launcher listens):

- TCP 3500 (default, configurable via `SERVICE_LAUNCHER_PORT`)

**Outbound** (Service Launcher connects to):

- TCP 3103 (Dashboard)
- TCP 3200 (Workspace API)
- TCP 3400 (Documentation Hub)
- TCP 3401 (Documentation API)
- TCP 3600 (Firecrawl Proxy)
- TCP 4005 (TP Capital)
- TCP 9000 (QuestDB)
- TCP 9090 (Prometheus)
- TCP 3000 (Grafana)

**Firewall Rules**:

```bash
# Allow inbound on 3500 (Dashboard access)
sudo ufw allow from 192.168.1.0/24 to any port 3500 proto tcp

# Allow outbound HTTP (health checks)
sudo ufw allow out 3103/tcp
sudo ufw allow out 3200/tcp
sudo ufw allow out 3400/tcp
sudo ufw allow out 3401/tcp
sudo ufw allow out 3600/tcp
sudo ufw allow out 4005/tcp
sudo ufw allow out 9000/tcp
sudo ufw allow out 9090/tcp
sudo ufw allow out 3000/tcp
```

### DNS Requirements

**None**. All services accessed via localhost or IP addresses.

### Network Connectivity

- **Localhost access** required for all monitored services
- **Internet access** not required (fully local operation)
- **VPN/Private network** optional (if monitoring remote services)

## Security Requirements

### Authentication

**None**. Service Launcher protected by:

- CORS restrictions (Dashboard + Docs origins only)
- Rate limiting (200 req/min)
- Local-only binding (not exposed to network)

### Authorization

**None**. All endpoints accessible to allowed origins.

### Encryption

**None**. All communication over HTTP (localhost only).

**For production** (if exposing to network):

- Use HTTPS reverse proxy (nginx, Caddy)
- Implement authentication middleware
- Use mTLS for service-to-service communication

## Docker Requirements (Optional)

### Docker Installation

```bash
# Install Docker
curl -fsSL https://get.docker.com | bash

# Add user to docker group
sudo usermod -aG docker $USER

# Start Docker service
sudo systemctl enable docker
sudo systemctl start docker

# Verify
docker --version
docker ps
```

### Docker Compose

```bash
# Install Docker Compose
sudo curl -L "https://github.com/docker/compose/releases/latest/download/docker-compose-$(uname -s)-$(uname -m)" -o /usr/local/bin/docker-compose
sudo chmod +x /usr/local/bin/docker-compose

# Verify
docker-compose --version
```

### Docker Socket Access

Service Launcher requires Docker socket access for container health checks:

```bash
# Verify socket exists
ls -la /var/run/docker.sock

# Test access
docker ps

# If permission denied, add user to docker group (see above)
```

## Upgrade Requirements

### From Previous Versions

**No data migration needed** - Service Launcher is stateless.

**Steps**:

1. Stop service
2. Pull latest code: `git pull`
3. Update dependencies: `npm install`
4. Run tests: `npm test`
5. Restart service
6. Verify health: `curl http://localhost:3500/health`

### Breaking Changes

**None yet** - Service Launcher is at version 1.x.

**Future breaking changes** will be documented in [Changelog](./changelog).

## Troubleshooting Requirements

### Common Issues

**Node.js version too old**:

```bash
# Check version
node --version

# Upgrade Node.js
sudo npm cache clean -f
sudo npm install -g n
sudo n stable
```

**Health check script not executable**:

```bash
# Make executable
chmod +x scripts/maintenance/health-check-all.sh

# Verify
ls -la scripts/maintenance/health-check-all.sh
```

**Docker socket permission denied**:

```bash
# Add user to docker group
sudo usermod -aG docker $USER

# Logout and login again, or run:
newgrp docker
```

**Port 3500 already in use**:

```bash
# Find process
lsof -ti:3500

# Kill process
lsof -ti:3500 | xargs kill -9

# Or change port
SERVICE_LAUNCHER_PORT=3501 npm start
```
